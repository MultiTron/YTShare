# 6. Проектиране

## 6.1 Функционални изисквания

Функционалните изисквания към приложението YTShare обхващат четири основни потребителски сценария, разпределени между мобилен и уеб клиент:

### Споделяне на видеоклипове в локалната мрежа

- Споделяне на YouTube линк от Android устройство към настолен компютър чрез системния Intent (`ACTION_SEND`).
- Автоматично откриване на активни YTShare Host устройства в локалната мрежа чрез mDNS/DNS-SD (`_http._tcp.`).
- Избор на целево устройство от списъка с открити компютри.
- Автоматично отваряне на споделения видеоклип в браузъра на настолния компютър.
- Извличане на метаданни за видеото (заглавие, thumbnail) чрез YouTube oEmbed API.
- Запазване на видеоклипа в локалната Room база данни с offline-first подход — при липса на интернет връзка видеото се маркира като `synced = false` и се синхронизира при възстановяване на свързаността.
- Преглед на история на споделените видеоклипове.

### Управление на потребителски профили

- Регистрация с email, парола, собствено и фамилно име чрез Firebase Authentication.
- Вход със съществуващ акаунт.
- Изход от системата.
- Преглед и редакция на потребителски предпочитания (тъмен режим, известия, tracking).
- Регистрация на устройства (hostname, IP адрес, порт) към потребителския профил.

### Система за приятелства

- Изпращане на покана за приятелство по email адрес.
- Приемане или отхвърляне на получена покана.
- Премахване на приятел от списъка.
- Преглед на списъка с приятели и чакащи покани.

### Чат съобщения

- Създаване на чат разговор при отваряне на приятелски профил (ако не съществува).
- Изпращане на текстови съобщения в реално време чрез WebSocket (STOMP протокол).
- Получаване на нови съобщения без презареждане — реалновременна доставка по канал `/topic/chat/{chatId}`.
- Push известия чрез Firebase Cloud Messaging (FCM) към устройствата на получателите, когато са офлайн или приложението е на заден план.
- Преглед на история на съобщенията в разговор.


## 6.2 Потребители

YTShare не разграничава потребителски роли (администратор/обикновен потребител) на текущия етап. Всички регистрирани потребители имат достъп до пълния набор от функционалности:

- Споделяне на видеоклипове от мобилното приложение.
- Управление на собствения потребителски профил и предпочитания.
- Изпращане/приемане/отхвърляне на покани за приятелства.
- Комуникация чрез чат с приятели.
- Преглед на история на споделените видеоклипове.
- Управление на регистрирани устройства.

Нерегистрираните потребители имат достъп единствено до landing страницата и страниците за вход/регистрация. Всички останали маршрути са защитени чрез `authGuard`.


## 6.3 Архитектура на проекта

Архитектурата на YTShare следва принципите на **многослойна клиент-сървър архитектура**, като включва четири независими компонента, комуникиращи помежду си чрез REST API, WebSocket и mDNS протоколи.

### Компоненти на системата

**1. YTShare.Android — Мобилно приложение (Kotlin / Jetpack Compose)**

Мобилният клиент изпълнява две основни функции:
- **Споделяне в LAN** — приема YouTube линкове чрез Android Share Sheet (`ACTION_SEND`), открива компютри в локалната мрежа чрез NSD (Network Service Discovery) и изпраща линка чрез HTTP GET заявка към десктоп услугата.
- **Социални функции** — комуникира с централния бекенд сървър чрез REST API (Ktor HTTP клиент) за управление на профил, приятелства и видеоклипове, и чрез WebSocket (Krossbow STOMP клиент) за реалновременен чат.

**2. YTShare.Host — Десктоп услуга (C# / ASP.NET Core)**

Минимален HTTP сървър, работещ на настолния компютър на потребителя. Слуша на порт 7296, регистрира се като mDNS/Bonjour сервиз (`_http._tcp.`) и при получаване на GET заявка на `/Share?link={url}` отваря линка автоматично в уеб браузъра. Може да работи като фонова Windows услуга.

**3. Backend — Централен сървър (Java 21 / Spring Boot 4)**

REST API сървър, обслужващ социалните функции на платформата. Отговаря за:
- Управление на потребители, приятелства, чатове, съобщения, видеоклипове и устройства.
- Автентикация чрез Firebase Admin SDK (верификация на Bearer токени).
- Реалновременна комуникация чрез WebSocket/STOMP за чат съобщения.
- Push известия чрез Firebase Cloud Messaging (FCM).
- Работа с PostgreSQL база данни чрез JPA/Hibernate.

**4. Frontend — Уеб приложение (TypeScript / Angular 20)**

Single Page Application (SPA), предоставящ уеб интерфейс за социалните функции. Комуникира с бекенда чрез REST API и WebSocket. Използва Firebase Authentication за управление на потребителските сесии.

### Комуникационни протоколи

| Връзка | Протокол | Описание |
|--------|----------|----------|
| Android ↔ Host | HTTP GET | Споделяне на линк в LAN (`/Share?link={url}`) |
| Android ↔ Backend | REST API (HTTPS) | CRUD операции за потребители, приятелства, видеа |
| Android ↔ Backend | WebSocket (STOMP) | Реалновременен чат (`/topic/chat/{chatId}`) |
| Frontend ↔ Backend | REST API (HTTPS) | CRUD операции |
| Frontend ↔ Backend | WebSocket (STOMP/SockJS) | Реалновременен чат |
| Android ← Backend | FCM Push | Push известия при нови съобщения |
| Android → Host | mDNS/DNS-SD | Автоматично откриване на устройства |
| Android → YouTube | oEmbed API | Метаданни за видеоклипове (заглавие, thumbnail) |
| Backend ↔ Firebase | Admin SDK | Верификация на ID токени |


## 6.4 Архитектура на приложението в Back-end

Сървърният модул е реализиран като Spring Boot приложение, следващо **слоеста архитектура** (Layered Architecture) с ясно разделение на отговорностите.

### Комуникация между слоевете

Комуникацията между слоевете следва еднопосочен поток:

```
HTTP заявка → Controller → Service → Repository → PostgreSQL
                  ↓              ↓
              DTO (Input)    Entity
                  ↑              ↑
              DTO (Output)   Mapper
```

**REST API слой (Controller)**

Контролерите (`@RestController`) приемат HTTP заявки, валидират входните DTO-та и ги предават на бизнес слоя. Всеки домейн има собствен контролер: `UserController`, `VideoController`, `FriendshipController`, `ChatController`, `MessageController`, `DeviceController`, `DeviceTokenController`, `UserPreferencesController`.

**Бизнес логика (Service слой)**

Service класовете инкапсулират бизнес правилата: валидация на данни, координация между repository-та, трансформация чрез MapStruct маппери и изпращане на WebSocket/FCM съобщения. Пример: `MessageService.createMessage()` запазва съобщението в базата, broadcast-ва го чрез `SimpMessagingTemplate` по `/topic/chat/{chatId}` и изпраща FCM push известия към устройствата на получателите.

**Достъп до данни (Repository слой)**

Интерфейси, наследяващи `JpaRepository`, осигуряват CRUD операции и допълнителни заявки. Примери: `MessageRepository.findByChatId()`, `FriendshipRepository.findByUserIdOrFriendId()`, `DeviceTokenService.getTokensByUserIds()`.

**Обектно картографиране (Mapper слой)**

MapStruct маппери трансформират между Entity и DTO обекти по време на компилация. Всеки домейн има собствен mapper: `UserMapper`, `VideoMapper`, `FriendshipMapper`, `ChatMapper`, `MessageMapper`, `DeviceMapper`, `DeviceTokenMapper`, `UserPreferencesMapper`.

**Сигурност и автентикация**

`FirebaseTokenFilter` прихваща всяка HTTP заявка, извлича Bearer токена от `Authorization` header-а и го верифицира чрез `FirebaseAuth.verifyIdToken()`. При успешна верификация създава `FirebaseAuthenticationToken` и го поставя в Spring Security Context. `SecurityConfig` конфигурира stateless сесии и определя публичните endpoints (`/api/public/**`, `/actuator/health`, `/ws/**`).

**WebSocket комуникация**

`WebSocketConfig` конфигурира STOMP broker с prefix `/topic` за broadcast съобщения и `/app` за application destinations. STOMP endpoint е `/ws` с SockJS fallback. `WebSocketAuthInterceptor` верифицира Firebase токена при WebSocket CONNECT.

## 6.6 Архитектура на база данни (PostgreSQL)

### Структура на БД и връзките между таблиците

Базата данни е организирана в 9 таблици, управлявани чрез Liquibase миграции. Всички таблици (с изключение на `chat_participants`) наследяват обща структура от `BaseEntity`: UUID първичен ключ (`id`), `created_at` и `updated_at` timestamps.

### Описание на таблиците и релациите между тях

**Таблица `users`**

Всеки запис съдържа уникален UUID идентификатор и основни данни за потребителския профил: `firebase_uid` (уникален идентификатор от Firebase Authentication), `email`, `first_name` и `last_name`. Полетата `firebase_uid` и `email` имат unique constraints.

Свързани релации:
- One-to-Many → `friendships` (като `user_id` или `friend_id`) — приятелски връзки.
- Many-to-Many → `chats` (чрез `chat_participants`) — участие в чат разговори.
- One-to-Many → `messages` (като `sender_id`) — изпратени съобщения.
- One-to-One → `user_prefs` (чрез `user_id`) — потребителски предпочитания.
- One-to-Many → `device_tokens` (чрез `user_id`) — FCM токени за push известия.

**Таблица `friendships`**

Описва връзки между двама потребители. Съдържа `user_id` (инициатор на поканата), `friend_id` (получател) и `status` — enum поле с възможни стойности `PENDING`, `ACCEPTED`, `REJECTED`.

Релации:
- Many-to-One → `users` (чрез `user_id`) — инициатор.
- Many-to-One → `users` (чрез `friend_id`) — получател.

**Таблица `chats`**

Представлява чат разговор между двама или повече потребители. Съдържа само audit полетата (`id`, `created_at`, `updated_at`), тъй като участниците са в отделна join таблица.

Релации:
- Many-to-Many → `users` (чрез `chat_participants`) — участници.
- One-to-Many → `messages` — съобщения в разговора.

**Таблица `chat_participants`**

Join таблица за Many-to-Many връзката между `chats` и `users`. Съдържа два foreign keys: `chat_id` и `user_id`.

**Таблица `messages`**

Съобщения в чат разговор. Всяко съобщение съдържа `content` (текстово съдържание), `status` (enum: `SENT`, `DELIVERED`, `READ`), `chat_id` (към кой чат принадлежи) и `sender_id` (кой го е изпратил).

Релации:
- Many-to-One → `chats` (чрез `chat_id`).
- Many-to-One → `users` (чрез `sender_id`).

**Таблица `videos`**

Споделени YouTube видеоклипове. Съдържа `title` (заглавие, до 128 символа), `url` (линк към видеото) и `thumbnail_url` (URL на миниатюрата).

**Таблица `devices`**

Регистрирани десктоп устройства (YTShare Host инстанции). Съдържа `host_name`, `ip_address`, `port` и `last_connected_to` (последно свързване). Свързва се с потребителските предпочитания чрез `user_preferences_id`.

Релации:
- Many-to-One → `user_prefs` (чрез `user_preferences_id`).

**Таблица `user_prefs`**

Потребителски предпочитания. Съдържа булеви полета `dark_mode`, `notifications` и `tracking`. Всеки потребител има най-много един запис (unique constraint на `user_id`).

Релации:
- One-to-One → `users` (чрез `user_id`).
- One-to-Many → `devices` (устройства, асоциирани с потребителя).

**Таблица `device_tokens`**

FCM (Firebase Cloud Messaging) токени за push известия. Съдържа `user_id`, `fcm_token` (уникален), `platform` (Android, Web и др.).

Релации:
- Many-to-One → `users` (чрез `user_id`).

## 6.7 Основни бизнес процеси

### 6.7.1 Регистрация на потребител

1. Потребителят попълва форма с 5 полета: собствено име, фамилно име, email, парола и потвърждение на паролата.
2. Angular frontend валидира формата (задължителни полета, съвпадение на паролите чрез `passwordMatchValidator`).
3. При валидна форма, `AuthService.register()` извиква `createUserWithEmailAndPassword()` от Firebase JS SDK.
4. Firebase Authentication създава потребителски акаунт и връща `UserCredential`.
5. `updateProfile()` задава `displayName` от собствено и фамилно име.
6. Чрез `switchMap`, `UserService.createUser()` изпраща POST заявка към бекенда (`/api/public/users`) с потребителските данни.
7. Бекенд `UserService` маппва `UserInputDto` → `User` entity чрез `UserMapper` и го запазва в PostgreSQL.
8. Потребителят се пренасочва към `/chats`.

### 6.7.2 Вход в системата

1. Потребителят въвежда email и парола в login формата.
2. `AuthService.login()` извиква `signInWithEmailAndPassword()` от Firebase JS SDK.
3. Firebase Authentication верифицира credentials-ите и връща `UserCredential` с ID токен.
4. `onAuthStateChanged` callback обновява `currentUser` signal-а в `AuthService`.
5. `authInterceptor` автоматично прикачва ID токена като `Authorization: Bearer <token>` header при всяка следваща HTTP заявка.
6. Потребителят се пренасочва към `/chats`.

### 6.7.3 Споделяне на YouTube видео от мобилно устройство към компютър

1. Потребителят натиска „Share" в YouTube приложението и избира YTShare.
2. `MainActivityCompose` получава Intent с `ACTION_SEND` и извлича URL-а.
3. `modifyLink()` премахва протокола (`https://`) и tracking параметрите от URL-а.
4. Приложението изпраща HTTP GET заявка чрез Volley към `http://{selectedHost}/Share?link={modifiedUrl}`.
5. YTShare.Host десктоп услугата получава заявката и отваря линка в браузъра чрез `Process.Start()`.
6. Паралелно, `saveLinkInfo()` извиква YouTube oEmbed API за метаданни (заглавие, thumbnail).
7. `VideoRepository.saveVideo()` запазва видеото — ако има интернет връзка, изпраща POST към бекенда (`/api/videos`); ако не, запазва в локалната Room база данни с `synced = false`.
8. При възстановяване на свързаността, `syncUnsyncedVideos()` синхронизира неизпратените видеоклипове.

### 6.7.4 Управление на приятелства

**Изпращане на покана:**
1. Потребителят въвежда email адрес в `AddFriendDialog`.
2. `ChatRepository.sendFriendRequest()` търси потребител по email чрез GET `/api/users/email/{email}`.
3. При намерен потребител, изпраща POST към `/api/friendships` с `status = PENDING`.

**Приемане/Отхвърляне:**
1. Получателят вижда поканата в секция "Pending Requests" на `FriendsScreen`.
2. При натискане на "Accept" — PATCH заявка с `status = ACCEPTED`.
3. При натискане на "Reject" — PATCH заявка с `status = REJECTED`.

**Премахване:**
1. Потребителят натиска бутона за премахване на приятел.
2. `AlertDialog` иска потвърждение.
3. При потвърждение — DELETE заявка към `/api/friendships/{id}`.

### 6.7.5 Изпращане и получаване на чат съобщения

1. Потребителят отваря чат с приятел.
2. `ConversationViewModel.load()` извиква `ChatRepository.getOrCreateChat()` — търси съществуващ чат или създава нов.
3. Зарежда историята на съобщенията чрез GET `/api/messages/chat/{chatId}`.
4. Абонира се за STOMP канал `/topic/chat/{chatId}` чрез `StompSessionManager`.
5. При изпращане на съобщение — POST към `/api/messages` с `content`, `chatId`, `senderId`.
6. Бекенд `MessageService.createMessage()`:
   - Запазва съобщението в PostgreSQL.
   - Broadcast-ва `MessageOutputDto` чрез `SimpMessagingTemplate` по `/topic/chat/{chatId}`.
   - Изпраща FCM push известия към устройствата на получателите чрез `FcmService`.
7. STOMP абонаментът на получателя получава съобщението в реално време и го показва в UI.
8. При излизане от разговора — `ConversationViewModel.unsubscribe()` прекратява STOMP абонамента.

### 6.7.6 Откриване и избор на устройства в локалната мрежа (NSD)

1. При стартиране на приложението (или при pull-to-refresh на Settings екрана), `NSDHelper` стартира NSD discovery за услуги от тип `_http._tcp.`.
2. Android NSD API изпраща mDNS multicast заявки в локалната мрежа.
3. При откриване на услуга, `onServiceFound` callback извиква `resolveService()` за получаване на IP адрес и порт.
4. `pickBestIp()` избира най-подходящия IP адрес (предпочитайки същия subnet чрез сравнение на мрежовия prefix).
5. `isValidIPv4()` валидира адреса чрез regex.
6. Откритите хостове се показват в списъка "Available Devices" на `SettingsScreen`.
7. Потребителят избира устройство чрез натискане на `HostCard`.
8. `SettingsViewModel.selectHost()` запазва избора в `SharedPreferences`.
9. При първо стартиране (IP = `0.0.0.0`), приложението автоматично избира първото открито устройство.
10. NSD discovery работи непрекъснато — при загуба на услуга `onServiceLost` я премахва от списъка, а при нови услуги ги добавя автоматично.
